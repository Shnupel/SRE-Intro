### 8. How the gateway finds the events service

The `gateway` service finds `events` by using the service name from `docker-compose.yaml`. Docker Compose creates a shared network called `app_default`, registers service names inside this network, and provides an internal DNS server at `127.0.0.11`. For this reason, when the application inside the gateway sends a request to `http://events:8081`, the name `events` is automatically resolved to the IP address of the `app-events-1` container.

At the time of my check, the name `events` resolved to `172.24.0.5`.

## Bonus Task. Trace a request across services

Commands:

```bash

sleep 5

RES=$(curl -s -X POST http://localhost:3080/events/1/reserve \
  -H "Content-Type: application/json" -d '{"quantity":1}')
RES_ID=$(printf '%s' "$RES" | python3 -c "import sys,json; print(json.load(sys.stdin)['reservation_id'])")
curl -s -X POST "http://localhost:3080/reserve/$RES_ID/pay"

```

Short explanation of the flow:

1. `events` created the reservation `dfdc87c2-e5e4-4186-a32f-4b3256b0d6c4` at `00:56:39.575`.
2. `gateway` logged the successful call to `POST /events/1/reserve` at `00:56:39.576`.
3. `payments` processed the charge and returned `payment_ref` at `00:56:39.646`.
4. `gateway` received the successful response from `payments` at `00:56:39.648`.
5. `events` confirmed the reservation and created the order at `00:56:39.651`.
6. `gateway` returned the final external response `POST /reserve/{id}/pay` with `200 OK` at `00:56:39.652`.

Estimated timing between the steps:

- from reservation to successful payment: about `71 ms`;
- from payment to confirmation: about `4-5 ms`;
- from confirmation to the final response to the client: about `1 ms`.

The total end-to-end time of the `reserve -> pay -> confirm` flow was about `76 ms`, if measured from the first internal request from gateway to `events` until the final response from gateway to the client.
